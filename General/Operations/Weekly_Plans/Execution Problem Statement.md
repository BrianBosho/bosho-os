# Execution Problem Statement

Source: [[../00 Inbox/Workbook q2 wk1 - Raw Archive|Workbook q2 wk1 - Raw Archive]]

## Problem statement captured from the workbook

The core problem is not a shortage of goals. It is the absence of a system that makes the current work, next work, milestones, and scheduling visible enough to execute.

## Questions recorded in the workbook

- How do I decide what to work on when I only have one hour?
- How do I think about progress I would have made if unplanned meetings had not taken over the day?
- What should I do with a small Saturday work window?
- How do I manage an app project that has no milestone structure or schedule?
- How do I structure the next two weeks for a paper revision?
- How do I see what exact work exists, how far along it is, and when to do it?
- How do I get through accumulated papers, books, blogs, and videos?

## Structural needs implied by the workbook

- Project plans
- Milestones
- Blocked-work tracking
- Scheduling
- Progress tracking
- A visible next action for each active project

## Why this note matters

This note is the clearest articulation in the workbook of the coordination problem the rest of the BrainDump is trying to solve.

## Problem breakdown

| Problem area | What is missing |
|---|---|
| Project visibility | no clear view of active projects and status |
| Milestones | no intermediate checkpoints |
| Scheduling | work is not placed into bounded time blocks |
| Next-action clarity | unclear what to do when time is short |
| Recovery from disruption | no method for re-scoping after meetings or interruptions |
| Backlog control | papers, blogs, videos, and books are accumulating without triage |

## Structural response checklist

- [ ] Give every active project a home page.
- [ ] Add milestones to projects that currently have none.
- [ ] Keep one visible next action per active stream.
- [ ] Track blocked work explicitly.
- [ ] Schedule work in bounded blocks, not vague intentions.
- [ ] Review backlog items and decide keep, defer, or drop.

## Control-page mapping

| Need named in this note | Where it should live |
|---|---|
| Project plans | project or research note |
| Milestones | project brief or execution note |
| Blocked-work tracking | weekly action board |
| Scheduling | weekly planning system |
| Progress tracking | project home pages and weekly review |
| Visible next action | every active note plus the weekly board |

## Decision rule

When there is only one hour available, the system should answer three things immediately:

1. Which active stream is most important right now.
2. What the next bounded task is.
3. What concrete output should exist at the end of that hour.
