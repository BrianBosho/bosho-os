# Spatial Localization of Visual Grounding: Experimental Protocol (Phases 0–8)

**Version:** 1.0
**Scope:** One-image case study, from sample selection through specificity analysis
**Prerequisite:** A working VLM grounding pipeline that can extract Layer-32 hidden states, projector outputs, and perform causal patch ablation with null-replacement intervention.

---

## Overview

This experiment tests a single question: for an object token generated by a VLM, does the model's internal causal support region align with the object's ground-truth spatial extent, or has the object's visual evidence become redistributed across patches?

The protocol proceeds in nine phases (0–8). Each phase specifies exactly what to do, what to compute, what to record, and what to check before proceeding. The protocol is designed for deterministic reproduction: every random seed, threshold, and hyperparameter is either fixed or explicitly varied.

### Terminology

Throughout this document:

- **Focal token** (`t*`): the generation step corresponding to the object noun (e.g., "table").
- **GT patch region** (`B_o`): the set of visual patches induced by projecting the COCO segmentation mask (or bounding box) onto the model's patch grid. This is NOT "the true support" at Layer 32 — it is the ground-truth object region projected into patch space.
- **Model-derived support set** (`S_{t*}`): the patches identified by the grounding pipeline as causally supporting the focal token at Layer 32.
- **Random matched set** (`R_k`): a control set of patches, equal in size to the comparison target, sampled uniformly at random from all patches.
- **Null pass**: a forward pass with a null/blank image (per the pipeline's null strategy), used to produce counterfactual hidden states and projector outputs for replacement during ablation.
- **Real pass**: a forward pass with the original image.
- **Control token** (`t_ctrl`): a non-object token from the same generation (e.g., a function word like "the" or "is"), used as a within-image negative control.

## Phase 0. Choose the Sample

### Objective

Select one image that gives the experiment the best chance of producing interpretable results. This is a case study — not a random draw.

### Selection Criteria

The image MUST satisfy ALL of the following:

1. Contains at least one clearly visible, non-occluded object.
2. Has a clean COCO annotation with a segmentation mask (preferred) or bounding box.
3. The object is likely to appear as a noun token in a caption generated by the VLM.
4. The object is medium-sized relative to the image: covers roughly 10–40% of the image area. Not tiny (too few patches), not dominant (GT region becomes nearly the whole image).
5. Minimal ambiguity: not a crowd scene, not heavy clutter.
6. Prefer an object whose expected token is a single subword token in the model's tokenizer. Check this before committing to the image.

### Procedure

1. Load the COCO validation set (or whichever annotated set you use).
2. Filter for images with segmentation masks where the mask area is between 10% and 40% of image area.
3. From the filtered set, select one image manually based on the criteria above.
4. Verify the expected focal token by running it through the tokenizer. If the object name fragments into multiple subword tokens, note this and plan the aggregation strategy (see Phase 1).

|-------|-------|
| Image ID | |
| Dataset | COCO val2017 (or specify) |
| Object category | |
| Bounding box coordinates (x, y, w, h) | |

| Object area as fraction of image area | |
| Why this image was chosen (1–2 sentences) | |

### Gate

Do NOT proceed if: the object covers less than 5% or more than 60% of the image, no segmentation mask is available and only a loose bounding box exists, or the expected focal token fragments into more than 3 subword tokens.

## Phase 1. Establish the Baseline Generation

### Objective

Run the model on the unmodified image and confirm that it produces the expected object token with non-trivial confidence.

### Procedure

1. Set the prompt to: `"Describe the image."`
2. Set decoding to greedy (temperature=0, no sampling).
3. Freeze all config: model weights, quantization, prompt template, null strategy. Record every parameter.
4. Run a single forward pass.
5. Extract the full generated text and the complete token sequence with probabilities.
6. Identify the focal generation step `t*` — the position where the object token appears.

**Subword aggregation rule:** If the focal object name spans multiple subword tokens (e.g., "giraffe" → ["gir", "affe"]), define `t*` as the FIRST subword token of the object. Record all subword token positions. For probability, report the product of conditional probabilities across the subword span: `p(object) = ∏_i p(subword_i | prefix)`. For log-probability, report the sum of log-probs. All subsequent causal interventions should measure the effect on this first subword token, with the multi-token product reported as a secondary metric.

7. Identify the control token `t_ctrl`. Choose a function word (e.g., "the", "is", "a") that appears BEFORE the focal token in the generated sequence.
8. Record the top-5 alternative tokens at position `t*`. Flag any synonyms of the focal object (e.g., "sofa"/"couch", "man"/"person").

### Record

| Field | Value |
|-------|-------|
| Full generated caption | |
| Complete token list (all positions) | |
| Focal token string | |
| Focal token ID(s) | |
| Focal token position `t*` | |
| Multi-subword? | Y / N |
| Baseline p(focal token) | |
| Baseline log p(focal token) | |
| Top-5 alternatives at `t*` (token, probability) | |
| Synonyms present in top-5? List them | |
| Control token `t_ctrl` string | |
| Control token position | |
| Baseline p(control token) | |

### Checks

-  The model actually emits the object token (it appears in the generated text).
-  The focal token probability is not trivially low (p > 0.05 as a rough guideline).
-  The caption is semantically sensible.
-  The control token is a genuine function word with high baseline probability.

### Gate

If the object token is absent from the generated caption, do NOT proceed with this image. Select a different image and restart from Phase 0.

## Phase 2. Validate Pixel-Space Necessity

### Objective

Establish that removing the object from the image at the pixel level actually disrupts the model's generation of the object token. This is the most basic causal claim. If it fails, the image is unsuitable.

### 2A. Pixel Object Removal

Create three edited versions of the image:

1. **Segmentation-mask gray fill:** Replace all pixels inside the segmentation mask with a uniform gray (RGB 128, 128, 128).
2. **Segmentation-mask Gaussian blur:** Apply a strong Gaussian blur (σ ≥ 20px) to the masked region only.
3. **Bounding-box gray fill:** Replace all pixels inside the bounding box with uniform gray.

For each edited image, run the SAME frozen pipeline (same prompt, same decoding, same config).

### 2B. Pixel Random Controls

Create K ≥ 3 random-mask controls:

1. Compute the pixel area of the segmentation mask.
2. For each control, generate a random contiguous rectangular region with the SAME pixel area as the segmentation mask.
3. Fill with gray (matching the gray-fill intervention).
4. Run the same pipeline on each.

### 2C. Run the Control Token Through All Conditions

For each of the above interventions, also record the probability of the control token `t_ctrl` at its original position. Object removal should specifically hurt the object token more than a function word.

### Record

For EACH of the (3 + K) interventions:

| Field | Value |
|-------|-------|
| Intervention type | seg-gray / seg-blur / bbox-gray / random-k |
| Object token still present in caption? | Y / N |
| If present: p(focal token) | |
| p(control token `t_ctrl`) | |

### Compute

- Δp_object(seg-gray) = baseline p(focal) − p(focal | seg-gray)
- Mean and std of random control effects on the object token
- Same for the control token

### Checks

-  Object removal (seg-gray) reduces the focal token probability substantially more than the mean random control.
-  Object removal effect on the focal token is substantially larger than its effect on the control token.
-  At least one of the three removal variants causes the object token to disappear from the caption entirely.
-  Random controls do not consistently eliminate the object token.

### Gate

If pixel-level object removal does NOT meaningfully reduce the focal token probability, this image is a bad sample. Return to Phase 0.

## Phase 3. Extract Grounding Artifacts

### Objective

Run the full grounding extraction pipeline on the original (unmodified) image to obtain all tensors needed for subsequent phases.

### Procedure

1. Run the real pass (original image) and null pass (null/blank image) through the model.
2. Extract and save the following tensors:

**Layer-32 quantities (for cosine comparison and support computation):**
- `H_real[t*]`: Layer-32 hidden state at the focal token position, real pass (shape: [hidden_dim])
- `H_null[t*]`: Layer-32 hidden state at the focal token position, null pass
- `H_real[t_ctrl]`: Layer-32 hidden state at the control token position, real pass
- `H_null[t_ctrl]`: Layer-32 hidden state at the control token position, null pass
- `V_real_L32`: Layer-32 patch states, real pass (shape: [num_patches, hidden_dim])
- `V_null_L32`: Layer-32 patch states, null pass

**Projector quantities (for intervention ONLY — never cosine-compare with L32 states):**
- `X_proj_real`: Projector output, real pass (shape: [num_patches, hidden_dim])
- `X_proj_null`: Projector output, null pass

**CLIP quantities (for optional later analysis):**
- `Z_clip_real`: CLIP encoder output, real pass (shape: [num_patches, clip_dim])
- `Z_clip_null`: CLIP encoder output, null pass

3. Verify tensor shapes and numerical sanity.

### Record

| Field | Value |
|-------|-------|
| Model ID and version | |
| Null strategy (e.g., blank image, mean image) | |
| Patch grid dimensions | (e.g., 24×24 = 576) |
| Focal token index `t*` | |
| Control token index `t_ctrl` | |
| H_real[t*] shape, dtype, norm | |
| V_real_L32 shape, dtype, norm range | |
| X_proj_real shape, dtype, norm range | |
| Any NaN/Inf values detected? | Y / N |

### Checks

-  All tensor shapes match expectations (patch count = grid_h × grid_w).
-  No NaN or Inf values in any tensor.
-  Projector outputs and Layer-32 states have different norms/scales (confirming they are in different spaces).
-  Real and null passes produce different tensor values (the null strategy is actually working).

## Phase 4. Build the GT Patch Region from the Object Annotation

### Objective

Project the COCO segmentation mask (or bounding box) onto the model's patch grid to obtain the ground-truth object region in patch space.

### Procedure

1. Determine the patch grid. For a typical VLM with 24×24 patches on a 336×336 input image, each patch covers a 14×14 pixel region. Record the exact mapping.

2. For each patch `j` (j = 0, ..., M-1 where M = num_patches), compute the occupancy fraction:
   ```
   a_j = area(patch_j ∩ segmentation_mask) / area(patch_j)
   ```

3. Construct two GT patch representations:

**4A. Hard GT patch set** using the segmentation mask:
   ```
   B_hard = {j : a_j >= θ}
   ```
   Run with TWO thresholds: θ = 0.25 and θ = 0.50. Report both throughout. The θ = 0.50 set is the primary hard set.

**4B. Soft GT occupancy map** using the bounding box:
   Keep `a_j ∈ [0, 1]` as a continuous weight for each patch. This is the primary localization reference for soft comparisons.

4. Compute the GT patch count at each threshold: `|B_hard(θ=0.25)|` and `|B_hard(θ=0.50)|`.

5. Sanity check: the GT region should cover a plausible fraction of total patches. If `|B_hard(θ=0.50)|` < 5 patches, the object may be too small. If `|B_hard(θ=0.50)|` > 0.6 × M, the object dominates the image.

### Record

| Field | Value |
|-------|-------|
| Patch grid size (e.g., 24×24) | |
| GT source (segmentation / bbox) | |
| Threshold θ = 0.25 patch count `|B_hard_0.25|` | |
| Threshold θ = 0.50 patch count `|B_hard_0.50|` | |
| GT patches as fraction of total (θ=0.50) | |

**Save visualizations:**
- Original image with segmentation mask overlaid
- Patch grid overlay with occupancy shading (heatmap of `a_j`)
- Binary patch grid at θ=0.25 and θ=0.50

## Phase 5. Compute Model-Derived Support Maps

### Objective

Extract the grounding pipeline's support maps for the focal token and the control token at Layer 32.

### Procedure

Compute the following for BOTH the focal token `t*` AND the control token `t_ctrl`:

**5A. State support map**

From the contrastive state delta at Layer 32, extract the per-patch support profile for the token:
- Raw delta row: `Δstate_j = f(V_real_L32[j], V_null_L32[j], H_real[t], H_null[t])` (per your pipeline's state grounding formula)
- Positive-only row: `max(Δstate_j, 0)` (zero out negative contributions)
- Normalized row: `Δstate_j / Σ_j max(Δstate_j, 0)` (normalize to sum to 1 over positive entries)

**5B. Decision support map**

Extract the decision-level support profile for the emitted token at position `t*`. This uses the decision grounding metric (e.g., `dec_dp_x_cvar` or whichever decision metric your pipeline computes per patch).

**5C. Causal support set**

Use the pipeline's causal support selection operator `Ω_{t*}` (top-mass or top-k selection):
- Identify the selected support patches
- Record the selection criterion (e.g., "top patches capturing 80% of total positive state support mass")
- Record the support set `S_{t*}` and its size `|S_{t*}|`

**5D. Activation magnitude baseline (spatial prior null)**

Compute a NON-CONTRASTIVE activation magnitude map: for each patch `j`, compute `‖V_real_L32[j]‖` (the L2 norm of the real-pass patch vector). Normalize to sum to 1. This represents what a naive "attend to high-activation patches" baseline would select, with no contrastive grounding logic. It serves as a spatial-prior null hypothesis.

### Record

For EACH of `t*` and `t_ctrl`:

| Field | Value |
|-------|-------|
| State support map (raw, save as array) | path |
| State support map (positive-normalized) | path |
| Decision support map (normalized) | path |
| Causal support set patch indices | list |
| Causal support set size `|S|` | |
| Causal selection criterion | description |
| Activation magnitude map (normalized) | path |

**Save visualizations (for `t*` only):**
- State support heatmap on the patch grid
- Decision support heatmap on the patch grid
- Causal support set highlighted on the patch grid
- Activation magnitude heatmap on the patch grid
- All four maps side by side for visual comparison

### Checks

-  Support maps are not trivially uniform (max / mean ratio > 2 for state support).
-  Some coherent spatial structure is visible in at least one support map.
-  State and decision maps have positive rank correlation (Spearman ρ > 0).
-  The causal support set is a proper subset of patches (not the whole grid).
-  The control token's support map differs qualitatively from the focal token's support map.

## Phase 6. Compare Support Maps to GT Before Any New Intervention

### Objective

Quantify how well the model-derived support maps overlap the GT object region. Pure localization comparison — no new model runs.

### Procedure

Compute for each combination of (support type) × (GT definition):

Support types: state, decision, causal, activation-magnitude-baseline
GT definitions: hard(θ=0.50), soft occupancy

**6A. Binary overlap metrics** (using hard GT set at θ=0.50 and binarized support):

For each support type, binarize the support map using the same selection criterion as the causal set. Then compute:
- **IoU** = |support ∩ GT| / |support ∪ GT|
- **Precision** = |support ∩ GT| / |support| (fraction of support patches inside GT)
- **Recall** = |support ∩ GT| / |GT| (fraction of GT patches covered by support)

**6B. Soft overlap / GT-mass capture** (PRIMARY localization metric):

Using normalized support weights `s_j` and soft GT occupancy `a_j`:
```
GT-mass capture = Σ_j (s_j × a_j) / Σ_j s_j
```
Measures: of all the support mass the model distributes, how much of it lands on GT-object patches?

**6C. Rank-based enrichment:**
```
enrichment = mean(s_j for j ∈ B_hard) / mean(s_j for j ∉ B_hard)
```
An enrichment > 1 means support is preferentially inside the object.

**6D. Support set decomposition:**

Decompose the causal support set `S_{t*}` into:
- `S ∩ B`: support patches that are INSIDE the GT region
- `S \ B`: support patches that are OUTSIDE the GT region

Record the sizes: `|S ∩ B|`, `|S \ B|`, and the fraction `|S ∩ B| / |S|`.

**6E. Control token comparison:**

Repeat metrics 6A–6C for the control token `t_ctrl` using the SAME GT region. The control token's support should overlap the focal object's GT region LESS than the focal token's support does.

**6F. Activation magnitude baseline comparison:**

Compute GT-mass capture for the activation magnitude map. If the grounding support maps have GT-mass capture no higher than this baseline, the grounding method is not adding spatial specificity beyond raw saliency.

### Interpretation Guide

- **High overlap** (IoU > 0.4, GT-mass capture > 0.5): Support tracks the object well at L32. Redistribution is modest.
- **Low overlap, but focal > control** (IoU < 0.2 but focal GT-mass capture > control GT-mass capture): Some object tracking remains, but substantial redistribution has occurred. Expected interesting case.
- **Low overlap, focal ≈ control**: Support maps may not be object-specific.
- **Grounding ≈ activation-magnitude baseline**: The grounding method is not adding spatial specificity.

Do NOT draw causal conclusions yet — overlap alone does not tell you which set is more important. That requires Phase 7.

## Phase 7. Layer-32 Causal Intervention Comparison

### Objective

Determine which patch set — model-derived support or GT object region — is more causally important for the focal token at Layer 32. This is the central test of the experiment.

### Critical Design Rule: Size Matching

All comparisons MUST be between patch sets of equal size. Ablating more patches causes more damage regardless of which patches you choose. Every comparison below uses size-matched sets.

### Procedure

**7A. Define the patch sets**

Let `n_S = |S_{t*}|` (support set size) and `n_B = |B_hard(θ=0.50)|` (GT set size).

**Round 1: At size n_B (GT-matched)**
- `S_topB`: the top-n_B patches from the support map by rank
- `B`: the GT patch set (exactly n_B patches)
- `R_k(n_B)`: K ≥ 3 random sets of n_B patches each

**Round 2: At size n_S (support-matched)**
- `S`: the full model-derived support set (exactly n_S patches)
- `B_topS`: the top-n_S GT patches by occupancy
- `R_k(n_S)`: K ≥ 3 random sets of n_S patches each

**Round 3: S∩B and S\B decomposition (at natural sizes)**
- `S∩B`: support patches inside the GT region
- `S\B`: support patches outside the GT region
- `R_k(|S∩B|)`: K ≥ 3 random sets of |S∩B| patches each
- `R_k(|S\B|)`: K ≥ 3 random sets of |S\B| patches each

**7B. Ablation procedure**

For each patch set listed above:

1. Start from the frozen model state (same as baseline).
2. At the projector injection point (where visual tokens enter the LLM), replace the selected patch vectors with their null-pass equivalents: `X_proj_real[j] ← X_proj_null[j]` for each `j` in the ablation set.
3. Run the forward pass from that point through the LLM.
4. Extract the focal token probability and log-probability at position `t*`.
5. Optionally: generate the full caption under ablation (useful for qualitative analysis).

**Total forward passes required:**

With K = 3: total = 1 + 5 + 5 + 8 = **19 forward passes** (plus 1 no-ablation re-run = **20 total**).

### Compute

For each condition:
- `Δlogp = logp_baseline - logp_cf` (positive means ablation reduced probability)
- `Δp = p_baseline - p_cf`
- `rel_drop = Δp / p_baseline`

### Interpretation Guide

**Round 1 & 2 (the main comparison):**

| Outcome | Meaning |
|---------|---------|
| Δ_S > Δ_B > mean(Δ_R) | Model support is more causally important than GT region. Evidence for redistribution. |
| Δ_B > Δ_S > mean(Δ_R) | GT region is more causally important. Support extraction is missing the true causal region. |
| Δ_S ≈ Δ_B, both > mean(Δ_R) | Both carry real causal information. Partial redistribution. |
| Neither Δ_S nor Δ_B > mean(Δ_R) | Attribution is weak. Neither set is specifically load-bearing at this granularity. |

**Round 3 (the decomposition):**

| Outcome | Meaning |
|---------|---------|
| Δ(S∩B) >> Δ(S\B), both > random | The support's causal power comes from object-interior patches. Context patches are passengers. |
| Δ(S\B) >> Δ(S∩B), both > random | The support's causal power comes from context patches outside the object. Strong redistribution evidence. |
| Δ(S∩B) ≈ Δ(S\B), both > random | Both interior and context patches contribute. |

### Checks

-  Baseline re-run matches Phase 1 probabilities exactly (determinism verified).
-  All random control sets are disjoint from S and B (no overlap with target sets).
-  Patch counts in each round are exactly matched.
-  Δlogp values are finite (no numerical overflow from ablation).

## Phase 8. Specificity Analysis

### Objective

Determine whether the causal effects observed in Phase 7 are specific to the chosen patch sets or are simply an artifact of ablating any set of that size.

### Procedure

**8A. Specificity margins**

For each targeted set T with matched random controls R_1, ..., R_K:
```
γ_T = Δ_T - mean(Δ_R_1, ..., Δ_R_K)
z_T = (Δ_T - mean(Δ_R)) / std(Δ_R)  [if K ≥ 3]
```

Compute specificity margins for:
- `γ_S`: support set specificity (Round 2)
- `γ_B`: GT set specificity (Round 1)
- `γ_{S∩B}`: inner support specificity (Round 3)
- `γ_{S\B}`: outer support specificity (Round 3)

**8B. Per-random-control detail**

Report each individual random control outcome. With K = 3, the spread estimate is noisy, so showing all points is essential.

**8C. Effect size comparison**

Compute the ratio:
```
γ_S / γ_B
```
If this ratio > 1, the support set has higher specificity than the GT set. This is the strongest version of the redistribution claim.

### Summary Statistics

| Metric | Value |
|--------|-------|
| γ_S (Round 2) | |
| γ_B (Round 1) | |
| γ_S / γ_B | |
| γ_{S∩B} (Round 3) | |
| γ_{S\B} (Round 3) | |
| Is γ_S > 0? | Y / N |
| Is γ_B > 0? | Y / N |
| Is γ_S > γ_B? | Y / N |
| Is γ_{S\B} > γ_{S∩B}? | Y / N |

### Interpretation Guide

| Outcome | Conclusion |
|---------|-----------|
| γ_S > 0, γ_B > 0, γ_S > γ_B | Both sets specifically load-bearing, model's support set MORE specifically important. Strongest redistribution evidence. |
| γ_S > 0, γ_B > 0, γ_S ≈ γ_B | Both are equally specific. Partial redistribution but GT still matters. |
| γ_S > 0, γ_B ≈ 0 | Support is specific but GT is not. Extreme redistribution — the original spatial location is irrelevant at L32. |
| γ_S ≈ 0, γ_B > 0 | GT is specific but support is not. The grounding method is failing to find the right patches. |
| Both ≈ 0 | Neither set is specifically important. |

Additional from the decomposition:

| Outcome | Conclusion |
|---------|-----------|
| γ_{S\B} > γ_{S∩B} > 0 | Context patches outside the object are the dominant causal contributors. Token is context-grounded. |
| γ_{S∩B} > γ_{S\B} > 0 | Object-interior patches are more important; moderate redistribution. |
| γ_{S∩B} > 0, γ_{S\B} ≈ 0 | Almost all causal power is from object-interior overlap. Redistribution is minimal. |

### Section 6: Preliminary Interpretation

State which of the three conclusions the evidence best supports:

**Conclusion A — Evidence for redistribution:** Pixel removal hurts, support/GT overlap is low, support ablation > GT ablation, support specificity > 0, and the decomposition shows context patches (S\B) contribute substantially.

**Conclusion B — Support metric failure:** Pixel removal hurts, GT ablation > support ablation, support does not beat random.

**Conclusion C — Context-grounded token:** Low overlap, both GT and support beat random, support includes neighboring context that makes semantic sense (the token depends on relational or scene-level information, not just the object's pixels).

### Section 7: Alternative Explanations to Consider

Before accepting any conclusion, explicitly address:

1. **Low-dimensional subspace encoding:** The object information might be encoded in a low-dimensional subspace of the L32 patch vectors rather than being spatially redistributed. Low overlap with GT wouldn't imply smearing — it would mean the information is compressed into a few dimensions of many patches.

2. **Attention sink artifacts:** Some patches may accumulate disproportionate attention regardless of content. If the support set heavily overlaps known attention-sink positions (e.g., the first patch, CLS-equivalent tokens), the causal effect may reflect disruption of attention routing rather than removal of object-specific information.

3. **Token position effects:** If the focal token appears early in generation, its support may be more diffuse simply because fewer textual constraints have accumulated.

### Section 8: Readiness for Phases 9–10

State whether the minimum success criteria are met:

-  Pixel object removal hurts more than matched random removal.
-  GT patch projection looks reasonable (5 < |B| < 0.6M).
-  Support map is not trivially uniform (max/mean > 2).
-  Support ablation beats matched random (γ_S > 0).
-  Support vs GT ordering is measurable and interpretable.
-  A coherent story can be told about the localization pattern.

### Random Seed Protocol

- Use a fixed random seed for ALL random set generation (random pixel masks in Phase 2B, random patch sets in Phase 7).
- Record the seed.
- The K random sets at each size should use seeds: `base_seed + 0`, `base_seed + 1`, ..., `base_seed + K-1`.
- This ensures reproducibility while ensuring the random sets are different from each other.
